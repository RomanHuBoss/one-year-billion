Ты — senior quant researcher, systematic trading architect, risk analyst, backend architect и технический аналитик с опытом проектирования production-grade торговых систем для crypto derivatives.

Исходная стратегическая идея:
- Стартовый капитал: 500 USDT
- Экстремальная цель: 1,000,000 USDT за 12 месяцев
- Рынок: Bybit Linear USDT Futures / USDT Perpetual
- Подход: regime switching
- Возможные режимы: no-trade, carry/funding, grid/mean reversion, pair stat-arb, breakout/momentum, de-risk
- Главная защита: hard risk engine
- ML-фильтр не должен сам открывать сделки, а только разрешать или запрещать уже сформированный setup
- LLM/news layer не должен управлять ордерами, размером позиции, плечом или risk limits

Критически важная установка:
Не надо делать красивое описание стратегии. Нужно найти слабые места, математические несостыковки, риск саморазрушения, скрытые допущения, ошибки проектирования, невозможные или экономически бессмысленные элементы, а затем дать исправленную инженерную спецификацию.

Требования к анализу:

1. Проверка математики цели
Проверь:
- Насколько реалистична траектория 500 → 1,000,000 USDT
- Какой ежедневный, недельный и месячный compounded return нужен
- Как просадки влияют на требуемую доходность восстановления
- Какое соотношение winrate / R:R / profit factor / turnover необходимо
- Насколько комиссии, funding и slippage делают цель труднодостижимой
- Где стратегия переходит из агрессивной в статистически саморазрушительную

Вывод должен быть честным:
- Что математически возможно
- Что практически маловероятно
- Какие условия должны совпасть
- Какие ограничения надо заложить, чтобы система не обнулилась быстро

2. Проверка торговой логики
Проверь каждый модуль:

A. Funding / Carry
- Есть ли реальный net edge после комиссии, funding, spread, slippage и hedge cost
- Возможен ли качественный hedge только внутри linear USDT futures
- Что делать, если hedge неполный
- Как оценивать beta hedge
- Как учитывать funding обеих ног
- Когда carry-модуль должен быть отключен
- Не является ли модуль слишком сложным для депозита 500 USDT

B. Grid / Mean Reversion
- Допустим ли grid только в RANGE
- Достаточна ли ширина диапазона после учета 2 комиссий, spread, slippage и buffer
- Нет ли скрытого усреднения против тренда
- Есть ли hard stop при выходе из диапазона
- Есть ли max inventory
- Есть ли запрет grid против сильного тренда
- Какие параметры grid разумны для счета 500 USDT

C. Pair Stat-Arb
- Достаточна ли корреляция
- Как считать rolling beta
- Как считать spread и z-score
- Как не получить directional exposure
- Что делать при partial fill одной ноги
- Как учитывать funding обеих ног
- Реализуем ли stat-arb на малом капитале
- Когда модуль лучше исключить из MVP

D. Breakout / Momentum Booster
- Есть ли реальный edge у breakout после costs
- Как отличить настоящий breakout от liquidity sweep
- Как учитывать объем, OI, funding, BTC regime, volatility expansion
- Где должен стоять stop
- Как считать position size
- Когда taker order допустим
- Как ограничить серию stop-loss
- Как не превратить booster в постоянный gambling-модуль

E. No-trade / De-risk
- Достаточно ли четко описаны условия запрета торговли
- Не должна ли система чаще выбирать no-trade
- Какие market states обязаны блокировать новые входы
- Как работает reduce-only режим

- какие действия запрещены;
- какие действия требуют подтверждения;
- какие действия должны логироваться;
- что делать, если backend вернул противоречивые данные.

Отдельно описать:
- как dashboard должен отображать no-trade;
- как dashboard должен отображать pending;
- как dashboard должен отображать active;
- как dashboard должен отображать blocked;
- как dashboard должен отображать de-risk;
- как dashboard должен отображать отсутствие ML verdict;
- как dashboard должен отображать риск-отклонение сделки;
- как dashboard должен отображать рассинхронизацию между локальной БД и биржей.

Специальное требование к Programmer-Ready Tickets:

Backlog задач должен быть полностью готов для передачи разработчикам.

Каждая задача должна содержать:

- ID задачи
- Название
- Тип задачи: backend / frontend / database / quant / ML / execution / risk / testing / DevOps / security
- Цель задачи
- Подробное описание
- Зачем это нужно
- Что создать или изменить
- Файлы / модули
- Входные данные
- Выходные данные
- Алгоритм реализации
- Edge cases
- Ошибки и исключения
- Логирование
- Audit trail
- Критерии приемки
- Обязательные тесты
- Зависимости от других задач
- Комментарий для разработчиков

Запрещено формулировать задачи слишком общо, например:
- “реализовать risk engine”
- “добавить dashboard”
- “сделать ML”
- “настроить API”

Вместо этого каждая задача должна быть атомарной и проверяемой.

Пример правильной формулировки:
“Создать модуль risk_engine/position_sizing.py, который рассчитывает qty, notional, effective_leverage и max_loss_if_stop на основе equity, stop_distance, risk_per_trade, min_qty, min_notional и instrument_specs. Сделка должна отклоняться, если min_notional вынуждает превысить допустимый риск.”

- как тестируется система;
- как frontend отображает состояние системы;
- какие задачи должны выполнить разработчики.

Документ не должен требовать дополнительных устных пояснений от автора стратегии.

Запрещенные форматы ответа:
